# SimpliXio Integrated Product System

## 1. Values-alignment audit
- Aligned: the strongest public promise is already present in README, onboarding, screenshots, settings, and automation docs.
- Aligned: newsletter, acquisition, and Discord flows are draft-first with redaction or approval gates.
- Aligned: iPhone, macOS, and Watch roles are separated: fast daily loop, deep workbench, decision glance.
- Drifting: older package/social copy still used broad "context engine" framing.
- Drifting: proof surfaces existed, but release notes, weekly review, product lessons, and feedback were not packaged as one Discord proof layer.
- Drifting: old App Store macOS screenshots were stale and screenshot automation was blocked by local Developer Tools permissions.

## 2. Product-shape audit
- Strong core: capture -> 3 priorities -> why -> action -> feedback.
- Risk: too many review concepts can make iPhone feel queue-heavy if exposed too early.
- Risk: integrations can turn the product into a broad signal dashboard unless every integration maps to better ranking or proof.
- Direction: keep iPhone focused on today's priorities and capture; keep macOS as the place for review, queues, and content work.

## 3. Strongest wedge recommendation
Recommended wedge:

> SimpliXio turns scattered thoughts, project noise, and open loops into 3 priorities and one next action.

Alternative wedges:
- Decide what matters now from your captured signals.
- Reduce open loops into one clear daily move.
- Turn project noise into priorities you can act on today.

Why the recommended wedge is strongest:
- It names the painful inputs.
- It names the concrete output.
- It is screenshot-friendly.
- It does not sound like a chatbot, task manager, or second brain.

## 4. Messaging rewrite recommendations
- Homepage/App Store/README one-liner: SimpliXio turns scattered thoughts, project noise, and open loops into 3 priorities and one next action.
- Subtitle: 3 priorities. Why they matter. What to do next.
- Onboarding opening: Capture the noise. SimpliXio gives you 3 priorities and one next action.
- Outreach line: I saw your work around project overload. SimpliXio turns scattered signals into 3 priorities and one next action.
- Product demo copy: Capture messy thoughts, let SimpliXio rank what matters, then act on one clear next move.
- Newsletter positioning: Public-safe lessons from real SimpliXio decision runs, with private context redacted.
- Discord/build-in-public phrasing: What changed, why it matters, what SimpliXio learned, and one clear CTA.

Screenshot headline set:
- Decide what matters now.
- Turn noise into 3 priorities.
- Know why it matters.
- Take the next action.
- See what was ignored.
- Private by default.

## 5. Product-core reorganisation plan
- Capture stays permissive and fast.
- Enrichment, scoring, ranking, and resurfacing stay backend-owned.
- Default iPhone surface stays Today-first: 3 priorities, why, next action, capture.
- macOS becomes the inspection workbench: Review Queue, Weekly Review, Decision Replay, recurring patterns, unresolved tensions, content candidates.
- Watch remains only top priority, next action, quick capture, and feedback.

## 6. Trust-layer improvements
- Keep "Private by default" visible in onboarding, settings, README, and newsletter surfaces.
- Keep "no autopublish" visible in newsletter, Discord, and acquisition docs.
- Keep private outbound as `needs_approval`.
- Keep public proof generated only from public-safe or redacted artifacts.
- Keep surfaced priorities explainable: why it surfaced, what helped confidence, what is missing.

## 7. Proof-layer improvements
- Package proof as artifacts, not claims:
  - SimpliXio Today
  - Weekly Review
  - Decision Replay
  - ignored signal counts
  - release notes
  - product lessons
  - newsletter/public-safe excerpts
- Never invent traction, users, revenue, or fake examples.
- Prefer fewer, better proof posts over daily noise.

## 8. Discord build-in-public plan
Recommended channels:
- `#announcements`
- `#release-notes`
- `#build-in-public`
- `#weekly-review`
- `#product-lessons`
- `#feedback`

Posting rules:
- Draft-only by default.
- Manual review before posting.
- One CTA per post.
- No private notes, raw logs, internal details, secrets, or noisy commit dumps.
- Every post must pass redaction and taste checks.

## 9. Integration priority recommendations
1. GitHub: project context, issue/PR/release awareness, release proof, builder relevance.
2. Notion: selected notes, specs, plans, decision memory, content enrichment.
3. Discord: curated proof, release notes, weekly updates, feedback prompts.
4. RSS: selective external signals and recurring themes.
5. Gmail: later-stage high-signal context only, not an inbox replacement.

## 10. Integrations to avoid or postpone
- LinkedIn scraping.
- LinkedIn website automation.
- Auto-sending private outreach.
- Full Gmail inbox processing.
- Slack or Calendar before GitHub/Notion/Discord/RSS prove value.
- Any integration that creates a feed instead of improving priority selection.

## 11. Acquisition alignment recommendations
- Lead with the wedge, not broad AI language.
- Prioritise warm signals: GitHub stars, OpenClaw interactions, content engagement, subscribers, people discussing overload and prioritisation.
- Automate discovery, scoring, summaries, and drafts.
- Keep private outreach approval-required.
- Measure signal quality: replies, saves, stars, trial starts, qualified conversations.

## 12. App Store and screenshot improvement plan
- Use current in-app screenshots only.
- First screenshots must show the product in use, not splash/promotional-only screens.
- macOS set should include Focus, Insights, Review Queue, Memory, Decisions, and Settings.
- First three messages should be:
  - Decide what matters now.
  - See what repeated.
  - Review only what matters.
- Regenerate macOS screenshots only after Developer Tools automation is enabled locally.

## 13. README / GitHub / OpenClaw credibility improvements
- README should open with the wedge and trust frame.
- Repo description should avoid "AI productivity platform" and "second brain."
- Release notes should explain what changed and why it helps the core loop.
- OpenClaw should act as proof that Pierre builds sharp, useful systems, not as a separate vague brand.

## 14. What to simplify or remove
- Remove broad public "context engine" framing from package/social copy.
- De-emphasise "marketing automation" language in favour of proof and trust loops.
- Keep queue/review depth off the default iPhone surface.
- Postpone broad integration breadth until GitHub/Notion/Discord/RSS have real signal.
- Keep backend sophistication out of public screenshots unless it explains an outcome.

## 15. Phased implementation plan
Phase 1:
- Canonicalise wedge, trust, and proof messaging.
- Add richer draft-only Discord proof outputs.
- Keep App Store screenshot generation strict against stale captures.

Phase 2:
- Tighten GitHub and Notion ingestion around ranking inputs.
- Add proof archive index from safe Discord/newsletter artifacts.
- Add weekly growth review tied to channel signal quality.

Phase 3:
- Add more advanced linking and resurfacing only after capture/ranking/proof loops are stable.
- Add Gmail only as selected high-signal context, not as an inbox.

## 16. First milestone to execute
- Publish this canonical product-system plan.
- Clean public copy that still says "context engine."
- Expand Discord proof drafts to include weekly review and product lessons.
- Run the strict weekly pipeline and manually review only the best draft per channel.

## 17. What to intentionally postpone
- Autopublishing.
- LinkedIn automation.
- Full inbox replacement.
- Large graph UI.
- Watch review surfaces.
- Any iPhone queue browser that weakens the daily decision loop.
